inline-affordances — Habit

The chat text pipe is an interface, not just a transcript. Text you emit is segmented and enriched into interactive objects at render time — a tool call without the tool, no server, no schema. The table is the first such affordance; write for it.

When

Composing a conversational reply whose content is a list-of-records: two or more items each described by the same small set of attributes — comparisons, option matrices, file / field / param listings, mappings, before/after, status rosters.

Not when authoring a KCD artifact file. Those are HTML and use the [data-kcd-table] faux-table ( governed by index-format ). This habit is about the chat turn — plain GFM in, rich object out.

Explanation

The chat text pipe is an interface, not just a transcript — text you emit is segmented and enriched into interactive objects at render time, a tool call without the tool (no server, no schema). It degrades cleanly: another agent reading your turn sees the same plain markdown table, perfectly legible, so the enrichment is a bonus for the human and never a tax on the AI reader. The table is the first such affordance, not the last; the reflex to train now is "tabular data → emit a table" so the surface gets used.

Rules

  • Reach for a table the moment the data is a list-of-records — don't prose-dump rows a reader has to reassemble in their head.
  • Standard GFM only: a header row, a |---| delimiter row, one row per item. Keep to ~2–4 columns; wider than that reads worse than prose.
  • The first column is the name / key — it renders mono, as the row's anchor. Put the identifying thing there.
  • Inline `code` around a path or identifier in a cell is fine and stays live ( a real path is still a clickable path token ).
  • One line per row — a GFM cell is single-line. A value that needs paragraphs is not a cell.
  • Write plain GFM, nothing else. Never hand-build an HTML table or [data-kcd-table] markup in a chat reply — the raw markdown is the input; the surface adds the styling and the rail.

Action

The chat render layer segments your reply and turns any GFM table into a first-class object — KCD styling plus a floating left rail of actions ( widen the table out of the narrow column into the right gutter, open it in its own Studio tab ). You get a structured, interactive affordance with zero tool calls, zero schema, zero validation round-trip: a tool call without the tool, riding the text pipe you're already writing into.

It degrades cleanly — another agent reading your turn sees the same plain markdown table, perfectly legible. The enrichment is a bonus for the human, never a tax on the AI reader.

The table is the first inline affordance, not the last — the same pipe will carry more as they are built. The reflex to train now is "tabular data → emit a table," so the surface gets used.
