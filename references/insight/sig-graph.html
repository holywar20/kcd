sig-graph — Reference

Overview

A SIG document (.sig — Starmind Insight Graph) is one JSON file holding a graph: two flat arrays, nodes and edges, edges referencing nodes by short stable id. It is dual-audience — the same file serves a human (who opens it in a studio tab and sees a spatial picture: shapes, colors, wires) and an agent (who reads the raw structure: a semantic type on every node, a named rel on every edge). Neither audience gets a lesser artifact; there is one file.

Authoring a graph is writing a JSON file. There is no API, no registry, no registration step. Write valid SIG JSON to any vault path ending in .sig and it is a graph — the extension is the identity, and opening that path in a studio tab routes it to the Insight Graph plug-in. Reading one is reading the file. This makes a graph a legitimate medium of communication: an agent can draw a picture for a human, or read the picture a human drew.

The governing split, and the one thing to get right: meaning lives in node.type and edge.rel; the layout block is pure look and carries no meaning. Geometry is partitioned into layout precisely so an agent can skip it whole — which is also why it costs nothing to make it rich. Be lavish in layout; be precise in type and rel.

The document

The whole shape. schema is always insight/1; kind names the insight tool that owns the doc (graph for the viewer that exists today):

{
  "schema": "insight/1",
  "kind": "graph",
  "nodes": [ … ],
  "edges": [ … ],
  "anchorMode": "specific"
}

anchorMode is optional and graph-wide: "specific" pins every wire end to a fixed anchor index; absent means nearest (wires auto-dock to whichever anchor faces the other node). Absent is the right default — omit the key unless pinning is the point.

The format is additive and forward-compatible, and the parse is tolerant by design: a blank or absent file is an empty graph (absence is not failure), unknown keys are carried rather than rejected, and a malformed document warns and falls back to empty rather than throwing — a studio tab never crashes on a bad doc. One consequence matters when authoring: a non-blank file that is not valid JSON is flagged malformed and the editor refuses to open it, so it can never be clobbered by a save. Emit valid JSON.

Nodes

A node is meaning plus a look. type is the semantic role — the agent's inference handle and an open, per-graph vocabulary (concept, fact, task, service, whatever the graph is about). It is not a render type and never a shape name. title renders as a block above the node; body is the text inside it.

{
  "id": "n1",
  "type": "concept",
  "title": "Economic thesis",
  "body": "Local models make the unit economics work.",
  "layout": { "x": 40, "y": 80, "w": 240, "h": 120, "shape": "rounded", "color": "amber" }
}

id is short and stable (n1, n2 …) and is what edges reference. layout is optional — a geometry-free node still renders, auto-placed on a 4-column fallback grid, and gains a real layout the first time a human moves it. Authoring explicit geometry is strongly preferred: you are drawing a picture, so place it.

type: "group" is the one reserved semantic type. A group node renders as a translucent container painted behind the ordinary nodes, with its title in the corner — use it to enclose a region of the graph. Size and position it to contain the nodes it groups; group membership today is spatial (what it visually encloses), not an explicit child list.

Edges

A wire is its own object with id-referenced endpoints and — the highest-value token in the whole document — a named relation:

{
  "id": "e1", "from": "n1", "to": "n2",
  "rel": "supports", "label": "feeds",
  "style": "curved", "dashed": false, "color": "gray",
  "headFrom": "none", "headTo": "chevron"
}

rel is required and open (depends_on, supports, derives …). Name it well: an unlabeled arrow is the single biggest loss of meaning in a diagram, and rel is what an agent reads. label is the human-facing caption drawn at the wire's midpoint and falls back to rel when absent — so omit it unless the caption should differ from the relation.

Each end carries its own arrowhead, independently: headFrom (source) and headTo (target), both defaulting to none. A conventional directed edge is headFrom: "none" + headTo: "chevron". dashed is orthogonal to style — a curved wire can also be dashed. fromAnchor / toAnchor pin an end to a specific anchor index; leave them out and the wire docks to the nearest anchor. An edge whose from or to names a missing node simply does not render, and deleting a node deletes every edge touching it.

The render vocabulary

These banks are closed sets — an unknown value degrades quietly rather than failing, but only these render as intended. This is the part that is otherwise undiscoverable without reading the surface.

Shapes (layout.shape, unknown → rounded) — generic geometry, ten of them, carrying no meaning whatsoever:

rectangle · rounded · pill · ellipse · diamond · triangle · hexagon · parallelogram · cylinder · document

Colors (layout.color, edge.color) — a ramp key, never a raw hex value. Unknown or absent → the element paints neutral. The ten canonical keys the picker offers are blue · cyan · teal · mint · lime · violet · fuchsia · red · orange · amber; the aliases silver · purple · green · pink · yellow · gray · grey also resolve. The hues are decorative and carry no meaning — the node's meaning is its type.

Fonts (layout.typeFont, layout.bodyFont — picked independently, absent → inter): inter (sans default) · plex (IBM Plex Sans) · grotesk (Space Grotesk) · lora (serif) · source (Source Serif) · mono (JetBrains Mono).

Wire styles (edge.style, default straight): straight · curved (a cubic Bézier) · elbow (orthogonal, two right-angle bends). Arrowheads (headFrom / headTo, default none): chevron · double · square · parallelogram · none.

Text alignment (layout.textAlign): center (default) or top — the body's vertical pull inside the shape.

Anchors — the perimeter dots wires attach to. On by default with one dot per side; anchors: false hides them entirely. Each edge of the node takes an independent count via anchorTop / anchorBottom / anchorLeft / anchorRight (absent → 1), spaced evenly along that side.

Authoring a graph

The recipe, in the order that avoids rework:

  1. Decide the two vocabularies first. What are the node types, and what are the edge rels? This is the graph's actual content; everything else is presentation. A graph with precise types and named relations is useful even unrendered.
  2. Write the nodes with real geometry. The default node is 200×96; the resize floor is 80×48. A comfortable column pitch is 240 and a comfortable row pitch is about 160. Origin is top-left, x grows right and y grows down, and the canvas is unbounded — negative coordinates are legal but starting near (40, 40) keeps things tidy.
  3. Lay it out so it reads. Flow left-to-right or top-to-bottom, consistently. Leave real gutters — a wire drawn between two nodes passes behind any node in its way (there is no obstacle routing), so overlapping boxes produce wires that appear to vanish. When a wire must cross a congested area, elbow is the clean answer.
  4. Wire it, naming every relation. Give direction with headTo: "chevron".
  5. Then decorate. Shape, color, font, alignment. This is free to the agent and valuable to the human, so spend it — but only after the meaning is right. Use color to group things visually, shape to distinguish kinds at a glance; just never let either be the only carrier of a fact, because the agent's read drops the whole block.
  6. Save to a path ending .sig, formatted as pretty JSON with tab indentation — that is what the editor writes, so matching it keeps diffs clean. A hand-authored example exercising every render property lives at sample.sig.

To read a graph, invert it: skip layout entirely and read the nodes as id · type · title · body and the edges as an edge list of from → to (rel). That projection is the whole document's meaning; the picture is the human's half.

Rules

Rule
Meaning goes in type and rel — never in shape or color. A diamond does not mean "decision" and amber does not mean "warning". If a fact is only encoded in the look, the agent's read loses it, because the projection drops layout whole.
Name every relation. rel is required and is the highest-value token in the document. An arrow that only means "related somehow" wastes the edge.
Color is a ramp key, never a hex. "amber", not "#f59e0b" — the renderer resolves the key onto the themed ramp. A raw hex resolves to nothing and paints neutral.
Be lavish in layout. Geometry is partitioned so it costs the agent nothing. Rich presentation is a feature, not indulgence — the human side should be excellent.
Ids are short, stable, and referenced. n1 / e1, unique within the document. Edges address nodes by id; an edge naming a missing node silently does not render.
The path is the registration. A file ending .sig is a graph — no registry entry, no API call. Do not invent a registration step.
Emit valid JSON. Parse is tolerant (blank → empty graph), but non-blank invalid JSON reads as malformed and the editor refuses to open it. Additive keys are safe; broken syntax is not.
Do not overlap nodes, and do not expect wire routing. A wire passes behind any node in its path. Leave gutters; reach for elbow in congested areas.
