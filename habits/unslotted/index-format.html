index-format — Habit

Inherited by all lenses via _lens_base. Defines the canonical shape of any navigational index (a nav-index artifact, or any What/Where/Why import block) in the project.

Why

Writing, modifying, or auditing any navigational index.

A navigational index is any surface whose primary purpose is navigation — pointing readers at other files. Examples: plans/nav-index.html, kcd/habits/nav-index.html, the Know and Do reference blocks in every lens, a catalogue under references/domain/. If a surface's job is to answer "where do I find X?" it's a navigational index.

Action

Structure each navigable section as a What / Where / Why faux-table — styled divs, never a real <table> (protocol §3). Each row is a data-kcd-slot="<kind>"; each cell is an addressable data-kcd-field. The kind is mandatory and set by POSITION — the row inherits the role of the block it sits in: link in a nav-index, reference in a Know References/Domains block, habit/contract/tool/ rule in the matching Do block; table-data for a plain non-linking row. A bare data-kcd-slot with no kind is invalid — the validator rejects it:

<div data-kcd-table>
  <div data-kcd-head><span>What</span><span>Where</span><span>Why</span></div>
  <div data-kcd-slot="link">
    <span data-kcd-field="what"  data-kcd-type="text">{label}</span>
    <a    data-kcd-field="where" data-kcd-type="path" href="{path}">{label}</a>
    <span data-kcd-field="why"   data-kcd-type="text">{use case}</span>
  </div>
</div>
  • What — the destination's name in plain language. Not the filename. ("Per-Widget API Surface", not "per-widget-api-surface".)
  • Where — a real <a> whose href is both the value and the navigation; link text is the file stem (or a meaningful short slug).
  • Why — one phrase capturing what the reader gets by clicking through. Bias toward purpose ("decompose persistence into per-widget endpoints") over content summary ("contains 6 phases and a table"). One sentence max.

Each row is exactly one navigable destination.

A 4th column is permitted only when an extra dimension is load-bearing for a specific index — e.g. What | Where | Why | Lens for a cross-cutting audit table. The first three columns and their semantics never change; only a trailing column may be added. Use sparingly (mirrors the same exception in lens_anatomy).

Multi-section indexes use one data-kcd-section per group, each with its own faux-table. The section names the group; the table holds the rows. Group by audience or status (e.g. Cross-lens / Complete / Retired), not by alphabet.

Format

  • Columns are set with the --kcd-cols custom property; order is cosmetic — the parser reads cells by data-kcd-field name, never by position.
  • The data-kcd-head row is human chrome (column labels); the parser ignores it.
  • Navigational slots carry no data-kcd-mode="suggested" — they are pure navigation, not dredge/inclusion.

Counter-examples (do not use)

  • Bullet list of links — drops the why, forces the reader to open each to learn purpose.
  • 2-column What | Where faux-table — same problem.
  • A real <table> — forbidden by protocol §3; use the styled-div faux-table.
  • Prose paragraph — not scannable, not atomically amendable.
  • Long Why text spilling to a second line — compress or move detail to the destination.

Explanation

The why column is the load-bearing one. Without it, a nav-index is just a directory listing — the reader has to open files to find what they need. With it, the reader can scan, decide, and click once. This is the difference between "I have a folder of plans" and "I have a navigable plan library." It compounds as the index grows.
