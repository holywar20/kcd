no-new-idioms — Habit

Why

about to create a new named abstraction the user did not ask for by name — a class, interface, base class, service, store, or connector/glue module whose existence is an implementation choice rather than the task itself

Action

announce it in the turn it's created — "New idiom: {Name} ({glue | connector | domain}). It exists to {one-line job}. Alternative considered: {fold into X / none obvious}." — then keep building

Explanation

The point is visibility, not permission — speed is preserved; silence is approval. Glue accumulates silently and compounds into conceptual bloat, but an announced idiom can still be collapsed cheaply before commit. One new class is fine; three unannounced connectors is an architecture decision nobody made.

Rules

  • the abstraction the task names is pre-approved — if the user asked for a FooService, build it, no announcement
  • everything else is announced in the turn it's created, not discovered at commit review; non-blocking — state it and keep building
  • the announcement collapses the data — what the idiom is, why the task forced it, and what the fold-in alternative was
