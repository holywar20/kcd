# Reference Categories

Framework meta-knowledge. Not auto-loaded. Read when adding a reference, or when
deciding where a piece of knowledge belongs.

---

## The rule

A KCD codebase has exactly one reference store: `_Claude/references/`. It is the
project-level store of truth — facts about the code, conventions, and domain that
lenses point at via their Know block.

References are **categorized**, not flat. Every reference file lives in a category
folder. A bare file at `references/` root (other than `index.md`) is a structural
violation — it has no declared category, so no one knows what kind of truth it is.

`kcd/` is the single source for everything that is *not* a reference: framework
meta-docs, templates, habits, procedures. If a document explains how the document
system works, it belongs in `kcd/`. If it states a fact about the project, it
belongs in a `references/` category.

---

## Standard categories

These four are the KCD-recommended top-level categories. A project may add more,
but should justify each addition — categories are a taxonomy, and a taxonomy with
twenty buckets classifies nothing.

```
_Claude/references/
  index.md                  catalog of every reference and its category
  style_guides/             how code should be written (formatting, naming, scope)
  patterns/                 reusable structural solutions (how to build an X)
  domain/                   business and data facts (schema, vocabulary, rules)
  codebase/                 generated navigation maps of the actual source
    {sub-project}/          one folder per code sub-project
```

| Category | Holds | Authored by | Volatility |
|---|---|---|---|
| `style_guides/` | Language and formatting conventions | Human, refined over time | Low |
| `patterns/` | "How we build an X here" structural templates | Human | Low |
| `domain/` | Schema, business vocabulary, invariants | Human + generated | Medium |
| `codebase/` | Route maps, controller maps, component maps | Generated by procedures | High — regenerated |

`codebase/` is special: it is the only category written by procedures rather than
by hand, and the only one expected to go stale between regenerations. It is split
per sub-project so a multi-repo codebase does not collide names (`backend/routes.md`
vs `frontend/routes.md`).

## IDT instantiation

```
_Claude/references/
  style_guides/   global.md  comments.md  php.md  vue.md  css.md
  patterns/       controller.md  sql.md  vue.md
  domain/         database.md  domain_concepts.md
  codebase/
    backend/      routes.md  controllers.md  data_map.md
    frontend/     routes.md  views.md  stores.md  api.md  components.md
```

Filenames are lowercase. The category folder already carries the "what kind of
thing" signal, so the filename only needs the specific subject.

## Where this is referenced from

- `kcd/kcd_framework.md` links here under "Reference categories."
- Lens Know blocks point at specific files inside categories, never at the
  `references/` root.
- `references/index.md` is the per-project catalog — it lists every file and is
  the human-facing "where is X" lookup.
