Frontmatter Schema — Framework

The frontmatter every artifact in the _Claude/ tree carries. This doc restates the contract for orientation; the locked authority is kcd-document-protocol §4 — where the two differ, the protocol wins.

Core principles

  1. No universal schema beyond the required four — the type's template is its schema. A lens looks nothing like a plan; that is correct, not drift. Each type's template (see templates) defines the field set it adds on top of the universal core.
  2. Frontmatter is a visible, addressable surface. It renders as a <dl data-kcd-frontmatter> whose every value is an inline-editable <dd data-kcd-field="…" data-kcd-type="…">. The same element is both the human-readable value and the machine address (protocol §1).
  3. Names are kebab-case. The name field follows the Skill standard — kebab, ≤64 chars, [a-z0-9-], no "claude"/"anthropic". Word separators within any name (including the leading-underscore framework scaffolds like _lens-base) are hyphens.
  4. On-disk ≠ dispatched. When context is compiled into a bundle, frontmatter and section scaffolding are stripped and Know/Care/Do are merged into one prompt block. Frontmatter is authoring/tooling metadata — its byte cost is paid only when an agent reads the raw file, never in the dispatched prompt.
  5. File-level, all-or-nothing validation. Every field declares a data-kcd-type mapped to a validator; if any required field is missing or fails, the whole file is discarded. A document is not active until it validates (protocol §1).

Status conventions

  • active = a live, validated artifact in place. Framework meta-docs and nav-indexes read directly from kcd/; any deployed copy is also active. active ⇒ the file has passed validation (protocol §1).
  • disabled in canonical kcd/ = a deployable seed — inert until deployed; the deploy script flips the deployed copy to active. Applies to the artifacts that are copied into the deployed tree: deployable lenses (including _lens_base), habits, and canonical generators/analyzers. disabled excludes the artifact from standalone loading (the compiler skips it; a link-follower reads it as "use the deployed twin").
  • composed = a live-in-place base — read directly and composed into its instances (e.g. _generator-base, _analyzer-base); never deployed and never run standalone.
  • deployed — the live state of a copied-and-flipped artifact in the deployed tree, where the deployment lifecycle tracks it distinctly from an in-place active.
  • draft · paused · complete · retired — the plan/work lifecycle values (a plan starts draft, ends complete or retired).
  • template leaves status blank — a scaffold is never run; the authored artifact receives a real status.

Artifact types

The type discriminator is one of the ten:

lens · plan · reference · framework · template · nav-index · habit · contract · generator · analyzer.

type is also a drift guard: a type↔location mismatch is a signal. (command, pipeline, index, doc, utility, and the category field are retired — superseded respectively by description-triggering, the dropped pipeline idiom, nav-index, framework, the deferred utility surface, and domain.)

Cross-cutting rules

  • Model convention: default claude-sonnet-4-6 (the reliable workhorse); downgrade to claude-haiku-4-5 only for genuinely trivial, fully-mechanical work. Declared per procedure in model.
  • disabled = ignored twice: behavioral (lenses skip) + mechanical (the compiler excludes).
  • Aggressive over passive: AI generates volume; default to retiring AI-authored artifacts. paused / explicit human action is the opt-out.
  • base kept on generator/analyzer/lens frontmatter as an explicit composition-chain link.
