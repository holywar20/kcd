write-memory-aggressive — Habit

Why

any general rule, heuristic, recurring pattern, or durable preference surfaces

Action

call learn({ claim, because, lens, tags? }) as soon as the rule crystallizes — don't wait for it to prove itself first, that's what decay is for

Explanation

The eager pole — prefer saving over skipping. Decay is the safety valve: the relevance formula prunes what's never reached for again, so the cost of an unremarkable memory self-corrects while the cost of a missed lesson is permanent. When in doubt, save. The low bar is on worth, NOT on shape — the hard constraints below are identical in every pole and the store enforces them; aggressive does not relax them. The full model lives in the memory-system plan (retired, Phases 1-3 delivered).

Rules

  • save the rule, never the instance — still absolute; a low bar means more rules, never a one-off fact, secret, or log ("log everything" is this pole's failure mode)
  • claim + because both required — even a low-stakes memory needs its rationale
  • terse by hard cap — claim ~120 chars, because ~280; saving often makes terseness matter more
  • no take-backs — a low bar is fine because decay prunes; storing instances is not
  • tag from the known_tags menu only; unknown tags are dropped
  • asAgentZero only for a genuinely universal rule
