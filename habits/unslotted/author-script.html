Why

About to write a one-off investigation, a recurring diagnostic, a build helper, or a test fixture — anything that runs beside the app rather than in it. Also when sorting or auditing the existing pile. The first question is always which species, because that picks the home and the lifecycle.

Action

place the script by species (build · diagnostics · probes · fixtures — never a flat dump), open every new one with the dated won't-ship header, and promote only on proven use

The homes — place by species, never flat-dump

Real logic lives in a repo's own scripts/, split by species. The dev-utilities tier holds only thin launchers.

  • scripts/build/ — durable build/deploy plumbing wired to npm (build-deploy, flush-dist, stage-node…). Long-lived; changes rarely.
  • scripts/diagnostics/ — recurring health checks and benches (smoke-*, verify-pass, eval-*, model-bench). Kept, stably named, dated, often deck-registered. The promotion candidates live here.
  • scripts/probes/ — disposable one-shot investigations (probe-*), one question each. Output goes ONLY to scripts/probes/out/ (gitignored) — never the repo root.
  • scripts/fixtures/ — test doubles and mock endpoints (mock-endpoint) the suite or a probe leans on.
  • _Claude/dev-utilities/*.js — the palette tier: a thin launcher that execSyncs a scripts/*.ts from the right cwd, registered as a row in commands.json. No logic here — the launcher only points at the source (see verify-pass.js).

Another repo's tooling (SDK, MCP) lives in that repo's scripts/, same species split. The folder encodes the species — don't restate it in a field.

The dated, won't-ship header

Every new script opens with a comment block carrying an authoring date, its species, the run line, and the won't-ship note. The date is the load-bearing part: a future agent culls the workshop by age, and an undated script is itself the signal — it predates this convention, so it is a prime cull candidate. We do not backfill dates onto legacy scripts; their datelessness already marks them.

/**
 * <one-line purpose>
 *
 * Authored: 2026-07-11 · Species: probe | diagnostic | build | fixture
 * DEV-ONLY — does not ship. The package build strips scripts/ and tests; shipped src/** must
 * never import from here. An undated script predates this convention → treat as cullable.
 *
 * Run: <how to invoke it>
 */

The promotion ladder

Most scripts are one-time junk; a few earn a longer life. Promotion is deliberate and follows proven use (rough-plug → find a real consumer → polish), never a hunch.

  • Junk (default) — a probe answers its question and is deleted, or left in probes/ to be swept later. Where most scripts end.
  • Diagnostic — a script that proves useful more than once graduates to diagnostics/: stable name, dated header, ready to re-run.
  • Dev-utility — a diagnostic worth one-click access gets a thin dev-utilities/*.js launcher and a commands.json row. The logic never moves — only a launcher is added.
  • Widget / key functionality — the rare script whose value is ongoing and visual (a model bench, a schema browser) is hoisted into an in-app surface (widget / studio tool). Not done yet — this is the endgame for the genuinely useful, and each such hoist is its own considered move, not an automatic step.

Rules

Rule
Scripts are dev-only and never ship. Shipped src/** must never import from scripts/; the packaging step strips scripts/ and tests. If a script's logic is needed at runtime, that logic belongs in src/, not imported out of the workshop.
Place by species; never flat-dump. A new file goes into build/ · diagnostics/ · probes/ · fixtures/ by lifecycle — not loose in scripts/.
Probes are disposable and write only to probes/out/. Never dirty the repo root with probe output. A probe is one question; when it's answered, delete it or leave it for the sweep.
Every new script carries the dated won't-ship header. Datelessness is not neutral — it marks a script as pre-convention and cullable. Do not backfill dates onto legacy scripts.
dev-utilities launchers are thin. A dev-utilities/*.js file only execSyncs the real scripts/*.ts; behaviour lives in the source, one place.
Cull on a sweep, not on sight. Deleting old scripts is the periodic cleanup pass (the plan-death cadence), never an inline action mid-task — nothing is deleted just because it looks stale today.
Promote on proven use. A script climbs the ladder only after a real, repeated consumer appears — not on a guess that it might be useful.
