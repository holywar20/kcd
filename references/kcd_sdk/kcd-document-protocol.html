KCD Document Protocol — Canonical Reference

Why

authoring, parsing, validating, or rendering any KCD document

This is stone. The locked specification for how a KCD artifact is encoded as an HTML document. Drift here is costly — it ripples through the parser, the validator, the renderer, the editor, and every authored and crystallized artifact. Change this only with an explicit ruling, never incidentally.

Substrate: standards-valid HTML — opens and navigates in any browser, rendered dark via one shared kcd.css, read mechanically by Starmind. No proprietary format. A small, closed set of data-kcd-* attributes carries all machine meaning.

1 · The addressing contract (the core law)

  1. Every machine value is an addressable field: data-kcd-field="<name>". The field's own content is the value — the element's text, or its href when the field is a link.
  2. One element, two duties. The same element surfaces the text to the human and is the machine address. No separate machine copy, no duplicated data-x="…" + visible span.
  3. No brittle dependencies. The parser reads only explicit data-kcd-* attributes and data-kcd-field values. Never element order, never grid position, never class names, never scraped free text. Anything not addressed is human chrome the parser ignores.
  4. Greppable, then fast. grep 'data-kcd-field="where"' finds every reference target in the corpus today; the explicit format lets faster structured tools be built tomorrow.
  5. File-level, all-or-nothing validation. Every field declares a data-kcd-type; each type maps to a tiny validator function. If any required field is missing or fails its validator, the whole file is discarded and ignored — never partially parsed. A document is not active until it validates. This is what keeps malformed data out of the system.
  6. Validators are shared. The per-type validator functions are a small composable library reused by SettingField — the same data-kcd-type vocabulary drives both document validation and the inline editor's controls.
  7. Reference implementation. The binary validator is kcd_sdk's KcdValidate (kcd_sdk/src/core/html/KcdValidate.ts), built on the Node-free HtmlTree reader and the KcdAddress grammar — pure JS/TS, no DOM dependency, file-level all-or-nothing exactly as above.

A field value's source by type: link fields (where, paths, urls) → href; all others → text content.

Structural rule — no empty containers. Include a region (or section) only when it carries content. Never emit an empty Know / Care / Do. An artifact uses only the structure it needs — most non-lens artifacts use few regions or none. (How much a single artifact should carry is left to judgment: a habit must stay atomic, not larded with fifty things. No mechanical cap yet.)

3 · Faux-tables (never a real <table>)

What/Where/Why-shaped idioms (References, Domains, Habits, Contracts, Parameters) render as styled divs, not <table>:

<div data-kcd-table>
  <div data-kcd-head><span>What</span><span>Where</span><span>Why</span></div>
  <div data-kcd-slot="reference" data-kcd-mode="on">
    <span data-kcd-field="what"  data-kcd-type="text">Framework concepts</span>
    <a    data-kcd-field="where" data-kcd-type="path" href="…/kcd_framework.html">kcd_framework</a>
    <span data-kcd-field="why"   data-kcd-type="text">the primitives this lens enforces</span>
  </div>
</div>
  • data-kcd-head is human chrome (column labels); the parser ignores it.
  • Each row is a component (data-kcd-slot / data-kcd-param); each cell is an addressable field.
  • Columns via the --kcd-cols custom property. Canonical column order: What / Where / Why.
  • Order is cosmetic — fields are read by name, so a reordered or renamed cell breaks neither parse nor render.
  • data-kcd-slot is canonical-only. A slot row's cells must be data-kcd-fields. Never put data-kcd-slot on a row you are not addressing — an unaddressed slot is invalid (the file fails).
  • data-kcd-slot carries its kind, by position. Every slot is data-kcd-slot="<kind>" from the closed set (§2); the parser reads it directly and never infers type from the where href. Kind follows the BLOCK: a References / Domains block yields references, Habits habits, Contracts contracts, Tools tools, Rules rules; any other block yields link (a row carrying a link) or table-data (a row with no link). Only the five dredge roles are routed / dredged — link and table-data ride their region as plain content, never as affordances.
  • Illustrative tables are chrome, not faux-tables. A descriptive table in prose (common in framework / reference docs) carries no canonical data — author it as a plain <table>. The parser ignores any <table>, and a <table> may never carry data-kcd-field / -slot / -param. Bright line: data-kcd-slot ⇒ canonical; <table> ⇒ chrome.

model is a parameter, not frontmatter. A generator/analyzer's model is the model parameter — the default model that seeds the agent's node in the work graph. Optional, and not yet implemented (no runtime reads it yet; we declare it now so authoring can start populating parameters). It is a data-kcd-param row whose default cell holds the model id (a connector route like remote.gemma31 is a valid default — default is freeform text, not a slug). Templates leave the default null.

Naming note. A leading _ on framework scaffolds (_lens-base, _lens-template) is a sort prefix, not part of the Skill name itself; word separators within the name are hyphens.

Templates. Templates are exempt from validation — scaffolds legitimately carry {placeholders} and the target type's own structure; template-aware validation is a deferred follow-up, not yet enforced. A template does not declare a target frontmatter field — the target type is inferred from the filename. utility is not a document type: it is declarative code (UtilityObject), not part of the template set. The index-type scaffold is filed as _nav-index_template.html (not _index_template.html), matching the index → nav-index rename.

5 · Audience & chrome (dual extraction)

The same surface yields two automatic extractions, chosen by consumer context:

  • Human read — the full visual: frontmatter panel, demos, styling. Optimized for the eye.
  • AI read — chrome stripped, optimized for brevity.

data-kcd-audience="human|agent|both" (default both) on any element is the strip control; human nodes are kept for the visual read and dropped from the AI read.

Chrome components (all human-audience by nature):

  • Living demo — data-kcd-chrome="demo": an interactive area (rendered in Starmind, placeholder in a plain browser).
  • Live widget — data-kcd-live="<id>": an embedded Starmind widget; inert placeholder elsewhere.
  • Bespoke vanilla JS — <script type="text/kcd-js" data-entry="<fn>">…</script>: the canonical insertion point for custom JS. The non-standard type means it does not execute in a browser (the script-exec gate stays closed outside Starmind, which extracts and sandboxes it). Source is shown to humans, stripped for the AI.

6 · Composable rules — habit-class and the cascade

A habit-class is a named mutual-exclusion group ("textual radio-buttons"): a set of behaviors where, by definition, only one may apply (e.g. subagent-policy = { no-subagents, always-use-subagents }). Classless habits are additive (all apply, union); classed habits are subject to the cascade below. Declared on a habit's own frontmatter (habit-class); mirrored on the including slot (data-kcd-habit-class), validated to agree.

Carriers (the first composition rule). Habits may be applied to exactly three things — a Lens, an Agent, or injected in a Workflow. No other artifact type carries habits.

Cascade priority — highest wins: Workflow > Agent > Lens. A lens-level habit is the base default; an Agent or a Workflow can override it. This is a one-dimensional override — like a primitive CSS cascade whose only specificity axis is the carrier level.

Resolution:

  • Within one carrier, including two habits of the same habit-class is a parser error — the file is discarded.
  • Across carriers, for each habit-class the highest-priority carrier that specifies a member wins; lower-level members of that class are overridden (dropped).
  • Classless habits never conflict — they union.

This is the spine of the composable-rules feature. Open: how an override surfaces in the UI (which habit won, which were overridden).

7 · Skill alignment (we are a superset)

KCD maps onto the Anthropic Agent Skills standard rather than redefining shared terms:

  • name + description adopt the SKILL.md field names and constraints (§4).
  • Dredge = progressive disclosure. data-kcd-mode="suggested" ⇒ full text always-loaded (Skill Level 1–2); on (the default, no attribute needed) ⇒ available-on-request, a routing row only (Level 3 / bundled resource); off ⇒ excluded entirely, not even a routing row.
  • No invocation command — KCD artifacts, like Skills, are description-triggered. This retires the whole command family: the lens !{name} command, the generator/analyzer #{name} task tag, and the session-log !{lens} token. CLAUDE.md may keep !lens / log shorthands as a Claude-level hook, but they are not KCD canonical and appear in no authored artifact.
  • Folder + bundled resources ↔ a lens folder + context/ + its slots.

8 · Rendering & theme

  • Dark by default. Light is a future productization theme — it overrides only the :root variable block in kcd.css; every component reads the variables.
  • One shared kcd.css styles every component by keying off the same data-kcd-* addresses the parser reads — styling and parsing share one contract.
  • Links navigate natively in any browser; chrome is inert outside Starmind.

8.1 · The stylesheet path (interim rule, 2026-07-01)

Canonical file: kcd.css. One copy, no duplicates. kcd/ is the project's canonical framework library — the right home for a shared, framework-wide asset.

Why this needs a rule at all. A KCD document is meant to open directly in a browser — file://, no server, no virtual root — so <link rel="stylesheet"> can only ever be a real filesystem-relative path. There is no /absolute that resolves the same way from every checkout, and a bare href="kcd.css" (what most of the corpus currently has) only works if a copy of the file sits in the same directory as the document — which, outside the original html-proposal origin trial, is not true anywhere. This is exactly the "dangling reference" class of bug the 2026-07-01 corpus audit surfaced.

The rule, for now: compute the relative path from the document's own directory up to _Claude/, then down into kcd/kcd.css — one ../ per directory level under _Claude/.

Document location (under _Claude/)Levels under _Claude/Correct href
lenses/_lens_base.html1../kcd/kcd.css
habits/{name}.html1../kcd/kcd.css
contracts/{name}.html1../kcd/kcd.css
lenses/{name}/{name}.html2../../kcd/kcd.css
references/domain/{name}.html2../../kcd/kcd.css
references/domain/orchestration/research/{name}.html4../../../../kcd/kcd.css

This is explicitly a stopgap, called out as such by the person who ruled it (2026-07-01): it still relies on whoever authors a document counting directory levels correctly by hand, which is exactly the kind of scut work a future authoring tool should absorb (a generator that stamps the correct relative path at deploy time, or a Starmind-side base-href rewrite when it renders these documents itself, removing the need for the raw file to carry a working link at all). Until that tool exists, an agent authoring or fixing a KCD document computes this path correctly by hand rather than copying the bare href="kcd.css" seen elsewhere in the corpus. The back-sweep to correct every already-deployed document's href to this rule is a separate, tracked pass — not implied by this section.

9 · Scope — what stays outside the substrate

The rule: every KCD primitive — lens, contract, habit, generator, analyzer, template, reference, plan, nav-index, note, how-to — and every document that describes or points at one, is authored as .html per this protocol. No exceptions, and no new primitive type is ever authored as .md. work/ is the sole grandfather clause: pre-2026-07-01 drafts left there are not back-converted, but anything new written to work/ is HTML from the start (see the plan contract's "every plan is HTML, at every stage" rule for the concrete case).

What is exempt — not KCD documents, so this protocol does not govern them:

  • Session, todo, and completed logs — _Claude/logs/session.md, _Claude/logs/{lens}/todo/todo.md, _Claude/logs/{lens}/completed/completed.md. Append-only breadcrumb lines, not dual-audience artifacts — see the session-log-aggressive habit (one of two poles of the session-logging mutual-exclusion slot — see §6).
  • Chat transcripts — _Claude/logs/chats/*.md. Raw session capture, not an authored document.
  • Raw generator/analyzer output — _Claude/audits/**, _Claude/reports/**, and a generator's own deployed context/ config/data files (e.g. an output-contract.md, a run-log.md). These are machine-written run records and config, the same kind as a registry — not something a human or agent reads as a document in its own right.
  • External standards — CLAUDE.md, SKILL.md, AGENTS.md. Conventions owned outside KCD (Claude Code, the Agent Skills standard); this protocol has no authority to reformat them.
  • Docs a generator writes into a target codebase it is not itself part of — e.g. safe_work/docs/front-end-component-map.md. That codebase's own documentation conventions govern, not this one.

Anything not on this list that is still .md outside work/ is drift, not a legitimate exception — flag it.

10 · Seeds — payload carried, not compiled

§9 puts a small set of files outside this protocol, CLAUDE.md and its siblings most of all. A seed is how a governed HTML artifact authors one of them: a literal payload block carried inside the document, extracted at deploy and written to a file this protocol does not govern. The artifact is the source and is never copied; the emitted file is a product. This is the sanctioned bridge across the §9 boundary — it lets an external-standard file be authored, versioned, and reviewed as a KCD document without claiming authority over its format.

Container. A seed is the §5 non-executing script idiom with a markdown type:

<script type="text/kcd-md"
        data-kcd-seed="claude"
        data-kcd-target="CLAUDE.md"
        data-kcd-mode="prepend">
# Heading

Markdown, verbatim.
</script>

The non-standard type means no browser executes or renders it, and script content is raw text — so the payload needs no HTML escaping. The single exception is a literal </script, which must be escaped.

  • data-kcd-seed="<host>" — required, a slug naming the agent host this payload is for (claude, codex, gemini). One document carries one block per host; the extractor selects by this key. Host keys are open, not a closed set — a new host is a new block, not a schema change.
  • data-kcd-target="<filename>" — required, the file written, resolved from the project root rather than the vault. A seed missing its target fails the file, per §1.5.
  • data-kcd-mode — prepend (default) or create-only.

Modes. create-only writes the whole file and never touches one that already exists. prepend maintains a managed block at the top of the target, delimited by <!-- kcd:begin --> and <!-- kcd:end -->; re-extraction replaces only what lies between the markers and leaves everything below them alone. That is what lets a vault deploy over a project whose CLAUDE.md already says things of its own, and re-deploy later without clobbering them.

A seed is not a slot. It carries no data-kcd-field cells, is never routed or dredged, and belongs to no faux-table. The two are unrelated primitives that happen to share an English word: data-kcd-slot is a routing row (§3), data-kcd-seed is a payload block.

Seeds never compile. The context compiler strips every seed block, exactly as it strips chrome (§5). A seed is deploy input, not agent context. A document carrying three host payloads would otherwise triple itself on the wire — and each payload is prose written for an agent that has not read the vault yet, which is precisely the wrong thing to hand one that has.

Resolved & still-open

Resolved (now stone): names are kebab-case (Skill standard; the leading _ on framework scaffolds is a sort prefix, not part of the name); regions are optional (§2 no-empty rule); the frontmatter field set (§4) is locked; the habit composition cascade (§6) is locked; author is universal, block form; domain (a special tag, may overlap) retires category; the template target field is dropped (inferred from filename); utility is not a document type (it is declarative code — UtilityObject); tables (§3) — data-kcd-slot canonical-only, illustrative tables are plain <table> chrome; a slot carries its kind explicitly and by block-position (dredge roles reference/habit/contract/tool/rule, domains folding into reference; non-dredge link/table-data), retiring href-inferred typing; templates are exempt from validation; the reference validator is pure-JS + binary; model is a parameter (§4); composed is a legal status; the command family is retired (§7); the index scaffold is _nav-index_template.html. The artifact-type set now includes note (a forward-looking observation — keeps its open / candidate sections, unlike a reference) and how-to (a procedural recipe — keeps its own step shape); observation is a legal (note) status; the non-HTML scope (§9) is locked — logs/chat-transcripts/raw-generator-output/external-standards/target-codebase-docs are the only legitimate .md, everything else is drift. The seed component (§10) is the sanctioned bridge to those exempt files — payload carried in HTML, extracted at deploy, stripped from compiled context; it is a distinct primitive from data-kcd-slot and shares nothing with it but the English word.

Retired — must NOT appear in any authored document: command / invocation (incl. the #{name} task tag and !{lens} log token — only CLAUDE.md keeps the !lens shorthand) · pipeline (superseded by Workflows) · index as a type name (→ nav-index) · category (→ domain) · the template target field · utility documents.

Still open (judgment for now): how much a single artifact may carry (atomicity vs. larding — no mechanical cap); UI surfacing of habit overrides (which won, which were dropped).
