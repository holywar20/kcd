Habit (template)

Scaffold note

A habit is a tiny, composable unit of behavior — a fragment of how an agent acts, not a task it performs. As of the four-field definition it is a fixed shape: four sections, in this order, so the same code can project every habit into one dense, instantly-recognizable form.

2026-07-13: this section was when until today — renamed to why to match the canonical What|Where|Why convention. It is now the SAME prose a lens's own Why cell can defer to by writing mode:habit instead of restating a condition — see the lens Care table's Why-cell grammar (always / habit / hand-written override). Nothing else about this field changed: it is still one terse trigger clause, still reads as "when {this}" in the projected grammar below.

  • why — the trigger. ONE terse clause: the precise moment or condition that fires this habit. Reads as "when {this}".
  • action — the behavior. ONE terse clause: the single step to take. Reads as "execute {this}". A don't-style habit omits this and leans on rules instead.
  • explanation — the rationale. The prose an agent gets when this habit is suggested (promoted from the manifest deck into injected context) — what makes the reflex worth following. Depth lives here; keep why/action terse and let this carry the weight.
  • rules — optional. Terse constraints, one per bullet. They project joined onto the second line, so keep each short.

How it projects (code-derived — never author the dense form by hand):

  • Deck (linked) — a terse manifest pointer: - {name} — {why} ({path}). By default this {why} IS this file's own why section (a lens's Care-table Why cell defaults to habit, meaning "defer to the habit's own why"); a lens may instead write hand-written prose in its own Why cell to override it for that one use. This is the exploration surface; the habit body does not ride.
  • Suggested (injected) — the dense two-liner from THIS file's four fields:
{name} — when {why}, execute {action}.
↳ {explanation} · {rules; joined}

Don't-style degrade (no action): {name} — when {why}: {rules}.

Copy to _Claude/kcd/habits/{class}/{verb-noun}.html (or habits/unslotted/ for a classless habit). Fill every {placeholder}, delete these scaffold notes. Frontmatter: type → habit; status → disabled (the deploy flips the copy to active); habit-class — optional slug naming a mutual-exclusion group (protocol §6), empty for an additive (classless) habit. Register it in habits/nav-index and wire it into the lens (or _lens_base) Do block that should inherit it — an unwired habit never runs.

Keep it atomic. If it needs phases, a human gate, or judgment, it is not a habit — it is a contract or an analyzer.
