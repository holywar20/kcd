Deploy Lens — Contract

The steps required to take a lens from the kcd library and activate it in a project.

C6 — deployment is moving to a script. Full copy + path rewrite, with file pathing 100% predictable. This contract is the spec the deploy script mechanizes, and the interim manual procedure until that script exists. The kcd source is never modified.

When

This contract activates whenever a session:

  • Receives #deploy-lens or "deploy a lens" / "activate a lens" / "wire up a lens"
  • Is asked to make a lens available for use in a project
  • Is preparing a new project for KCD-based development

Artifact Format

A deployed lens is a folder _Claude/lenses/{name}/ containing {name}.html plus a context/, sourced from _Claude/kcd/lenses/{name}/{name}.html.

Frontmatter (after deployment):

---
type: lens
status: active
command: "!{name}"
todo_path: _Claude/logs/{name}/todo/todo.md
completed_path: _Claude/logs/{name}/completed/completed.md
---

Rendering conventions: Per _lens_base — paths are links; backticks-around-paths reserved for pattern paths, fenced code, and quoted speech; blank line before every table.

Lifecycle

Phase 0 — Check Base

Trigger: Any lens deployment is requested.

Together: Check whether _Claude/lenses/_lens_base.html exists. If it does, proceed to Phase 1. If not, deploy _Claude/kcd/lenses/_lens_base.html first — copy it to _Claude/lenses/_lens_base.html, set status: active. Base has no CLAUDE.md registration, but it is path-rewritten in Phase 3 like every other lens — there is no exception. A deployed artifact never links back to kcd/ for any artifact that has a deployed twin (the previous design left base pointing at _Claude/kcd/habits/…, but those canonical copies ship status: disabled, so an agent following the link read a disabled template instead of the active deployed habit — the exact drift this rewrite exists to prevent). Base still deploys its descendant habits and contracts per Phase 2.5 (write-approval, work-routing, logging, plan). Once base is deployed, proceed to Phase 1.

Standard: _Claude/lenses/_lens_base.html exists and is status: active before any other lens is deployed. Base's descendant habits and contracts exist at their deployed locations. Base is deployed once and never redeployed unless explicitly requested.

Phase 1 — Verify Source

Trigger: A lens deployment is requested.

Two paths depending on lens origin:

Path A — kcd-sourced lens (the lens exists at _Claude/kcd/lenses/{name}/{name}.html): Confirm the kcd source carries status: disabled. Read it and check anatomy compliance: Know / Care / Do structure present, References table uses What | Where | Why headers, no paths hardcoded to another project (portability boundary). Surface any anatomy failures before proceeding. A lens with status: disabled and clean anatomy is ready to deploy via Phase 2.

Path B — in-place authored lens (the lens was drafted directly at _Claude/lenses/{name}/, no kcd source exists): This is the common case for project-specific lenses. Skip the kcd source check. Read the deployed lens and verify anatomy compliance: Know / Care / Do structure present, References table headers correct, deployed habit and contract links point at _Claude/habits/ and _Claude/contracts/ (not _Claude/kcd/), no paths hardcoded to another project. Surface any violations before proceeding. If the lens is already status: active and anatomy is clean, proceed directly to Phase 2.5.

Standard: No deployment proceeds against a lens with anatomy violations. The origin path (A vs B) determines whether a copy step is needed — it does not change the anatomy requirements.

Phase 2 — Copy and Activate

Trigger: Phase 1 passes.

Together: Copy the lens folder _Claude/kcd/lenses/{name}/ to _Claude/lenses/{name}/. Set status: active in the deployed copy. The kcd source retains status: disabled — never modify the kcd original.

Populate todo_path and completed_path in the deployed frontmatter per the artifact format (_Claude/logs/{name}/todo/todo.md and _Claude/logs/{name}/completed/completed.md). Create the empty directories _Claude/logs/{name}/todo/ and _Claude/logs/{name}/completed/ and seed the two files with frontmatter only (type: log, status: active). Folder-plus-file is the v1 shape; breakout to many files is a forward path, not a v1 concern.

Standard: The deployed folder exists at _Claude/lenses/{name}/. The kcd source is unchanged. The two log files exist and carry valid frontmatter.

Phase 2.5 — Deploy Descendant Habits & Contracts

Trigger: Phase 2 complete.

Together: Read the deployed lens's Do > Habits and Do > Contracts tables. For each row pointing into _Claude/kcd/habits/ or _Claude/kcd/contracts/, copy the source file to its deployed location (_Claude/habits/{name}.html, _Claude/contracts/{name}.html). Set status: active in each deployed copy. If a deployed copy already exists, skip with a one-line surface to the user; never overwrite silently.

Standard: Every habit and contract the lens depends on exists at its deployed location with status: active. The kcd sources remain disabled and unchanged.

Phase 3 — Rewrite Paths

Trigger: The deployed copy exists (Path A) or anatomy check passes (Path B).

The governing rule. A deployed artifact never links back to _Claude/kcd/ for any artifact that has a deployed twin. The reason is separation of lifecycles: everything under _Claude/kcd/ is the locked canonical library (and its lens/habit/contract copies ship status: disabled); everything a deployed lens operates against lives in the parallel deployed tree, is status: active, and drifts as the user edits it. If a deployed agent follows a link into kcd/, it reads the frozen, disabled template instead of the live copy it actually runs on. The canonical path authority every rewrite conforms to is deployment_schema.canvas; future test scripts validate against it.

The twin map — rewrite each kcd/ source path to its deployed destination:

Artifact type Canonical source (kcd/) Deployed destination Rewrite?
habit _Claude/kcd/habits/{name}.html _Claude/habits/{name}.html Yes — always
contract _Claude/kcd/contracts/{name}.html _Claude/contracts/{name}.html Yes — always
lens (incl. _lens_base) _Claude/kcd/lenses/{name}/{name}.html _Claude/lenses/{name}/{name}.html Yes — always
framework doc _Claude/kcd/kcd_framework.html (no deployed twin) No — canonical-only authority
structural doc _Claude/kcd/docs/{lens_anatomy,frontmatter_schema,…} (no deployed twin) No — canonical-only authority
template _Claude/kcd/templates/_{type}_template.html (no deployed twin) No — canonical-only authority
type-base _Claude/kcd/{generators,analyzers}/_{type}_base.html (no deployed twin) No — canonical-only authority

Worked examples (the link's target is what changes; the display label is untouched):

BEFORE  | run-report | [run-report](_Claude/kcd/habits/unslotted/run-report.html) | … |
AFTER   | run-report | [run-report](_Claude/habits/unslotted/run-report.html)     | … |

BEFORE  | Plan | [plan](_Claude/kcd/contracts/plan.html) | … |
AFTER   | Plan | [plan](_Claude/contracts/plan.html)     | … |

BEFORE  *(in a habit body)* …inherited via [`_lens_base`](_Claude/kcd/lenses/_lens_base.html).
AFTER   *(in a habit body)* …inherited via [`_lens_base`](_Claude/lenses/_lens_base.html).

UNCHANGED  *(canonical-only — no deployed twin exists)*
           | Framework concepts | [kcd_framework](_Claude/kcd/kcd_framework.html) | … |
           | Lens template | [_lens_template](_Claude/kcd/templates/_lens_template.html) | … |

Procedure. Rewrite every _Claude/kcd/habits/…, _Claude/kcd/contracts/…, and _Claude/kcd/lenses/… link target in the deployed artifact to its deployed destination per the twin map — in the lens body and in any habit/contract bodies co-deployed in Phase 2.5 (those commonly carry a back-link to _lens_base). Leave the four canonical-only rows untouched: there is no deployed copy to point at, so they remain the legitimate residual kcd/ references. References already within lenses/, references/, work/, etc. are vault-root-relative and survive the move unchanged.

_lens_base is not an exception — rewrite its habit and contract links exactly like any other deployed lens. (A prior version of this contract exempted base; that exemption was the source of a live drift bug and has been removed.)

(Path B lenses may already have correct paths — verify before rewriting. This step is a no-op if all links are already at deployed locations.)

Standard: Every link in the deployed artifact resolves to a file that exists. Every habit, contract, and lens link with a deployed twin points at the deployed tree, never _Claude/kcd/; the only surviving kcd/ links are to canonical-only material (framework docs, structural docs, templates, type-bases). Broken links are surfaced to the user — do not deploy an artifact pointing at missing files without flagging it.

Phase 4 — Register

Trigger: Paths verified.

Together: Two writes.

4a — Lens nav-index. Append the lens row to _Claude/lenses/nav-index.html. If the file does not exist, create it from _nav-index_template with a Deployable lenses section (mirroring the shape of kcd/lenses/nav-index), then append. This write is not approval-gated — the nav-index is a structural projection over the deployed roster.

Row shape — a What / Where / Why faux-table slot (per the index-format habit):

<div data-kcd-slot="link">
  <span data-kcd-field="what"  data-kcd-type="text">{name}</span>
  <a    data-kcd-field="where" data-kcd-type="path" href="_Claude/lenses/{name}/{name}.html">{name}</a>
  <span data-kcd-field="why"   data-kcd-type="text">{one-line purpose from the lens Purpose section}</span>
</div>

4b — CLAUDE.md. Append the lens row to the Lens Index table in CLAUDE.md:

| What | Where | Why |
|---|---|---|
| `!{name}` | [{name}](_Claude/lenses/{name}/{name}.html) | {one-line purpose from the lens Purpose section} |

If CLAUDE.md does not exist, create it with at minimum a one-line project header and the Lens Index table (header + this one row). CLAUDE.md is approval-gated — per the write-approval habit family, lens_crafter drafts the change to _Claude/work/lens_crafter/plans/ first and presents the draft for explicit approval before applying.

Standard: The lens appears in both indexes. Both one-line purposes are drawn directly from the lens's Purpose section — not paraphrased.

Phase 5 — Verify

Trigger: Registration complete.

Together: Do a final read of the deployed lens. Confirm: status: active, link convention followed per _lens_base (paths are links; backticks only for pattern/code/quoted-speech), all tables have blank lines before them, Working Space paths reference _Claude/work/{name}/.

Standard: The lens is ready to load. Announce the command (!{name}) and the working space location to the user.

Standards

  • kcd is the only valid source. Never copy from another project's _Claude/lenses/.
  • kcd source is never modified during deployment. The kcd copy stays disabled.
  • A deployed artifact never links back to kcd/ for any artifact with a deployed twin. Habit, contract, and lens links resolve to the deployed tree — full stop. This is the load-bearing rule of Phase 3, restated here because it is the one most easily violated: a stray kcd/habits/… link reads a disabled seed instead of the live deployed copy, and the agent silently runs on the wrong file. The only surviving kcd/ links are to canonical-only material with no deployed twin (framework docs, structural docs, templates, type-bases); those are referenced in place. Framework/structural docs carry status: active and type-bases carry status: composed — either way the file itself confirms the link is live and correct to follow; templates are status:-blank scaffolds. (composed = live in place but never run standalone — see the frontmatter schema.)
  • CLAUDE.md is approval-gated. The index update requires explicit user sign-off.
  • Broken paths block deployment. A lens pointing at missing files is flagged, not silently deployed.
  • Obsidian rendering conventions are non-negotiable. Blank lines before tables; markdown links throughout.

Edge Cases

  • Lens already deployed (status: active at _Claude/lenses/): Surface to user — confirm whether to overwrite or skip.
  • Referenced file doesn't exist (e.g. a doc or template the lens points into kcd/): Flag the broken link; deploy the lens anyway with a todo noting the gap. Habits and contracts referenced by the lens are co-deployed per Phase 2.5 and should not normally appear in this case.
  • kcd source has hardcoded paths from another project: Surface during Phase 1 anatomy check; fix paths before proceeding.
