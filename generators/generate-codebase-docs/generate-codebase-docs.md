# generate-codebase-docs — Generator (canonical)

*Generic across any codebase. Reads a project's source files and emits a set of navigation
references designed to give an agent a fast, navigable attack surface — enough to plan and
execute substantial work without reading raw source. An **output-contract** (deployed at
`context/output-contract.md`) declares which strategies run, what sources they read, and
where output goes. The canonical defines the strategy shapes and execution model; the deployed
copy solves all project-specific values.*

Base rules: [_generator_base](_Claude/kcd/generators/_generator_base.md) — composition model,
requirement resolution, fail behavior, output conventions, modifiers.

## Output Strategies

*Five strategies. Each is optional — enabled or disabled in the output-contract. The
**write-if condition** is a natural language predicate evaluated at deployment time and
recorded in the contract. The running generator reads the result and never re-evaluates.
Judgment lives in the deployment session, not in the run.*

**Canonical run order:** api-surface → route-map → folder-index → domain-map →
composition-graph. Running api-surface first surfaces the external call layer, which informs
every subsequent strategy — it establishes integration points and dependency context before
the other docs are written.

---

### `api-surface`

**Write-if:** The codebase makes external calls from an organized call layer — a module,
service, or set of functions whose job is to reach outside the application's own process and
fetch or send data.

*This is outbound calls only — not what this system exposes, but what it calls elsewhere. It
is the anti-duplication index: an agent writing new feature code checks this doc before adding
a call that may already exist. Running this strategy first benefits the rest of the run
because the external call map reveals integration points, dependency patterns, and naming
conventions that make every other doc more precise.*

**Source:** The external-call layer(s) declared in the contract. Covers HTTP clients, fetch,
axios, curl-equivalents — any mechanism that exits the process. May be a glob over a dedicated
`api/` or `services/` folder, or a single file that centralizes all calls.

**Output:** One reference doc. Organized by module or file. Each section covers one source
file's calls.

```
# API Surface — Outbound Calls
*Generated by #generate-codebase-docs (api-surface) on {date}. Flush-and-fill.*

## {ModuleName} — {source-path}
| Function | Method | Endpoint | Returns | Notes |
|---|---|---|---|---|
| {fn(params)} | GET/POST/… | {url pattern} | {return shape} | {optional} |
```

*If a module has zero identifiable external calls, include its section header and note
"no external calls identified — verify." Do not silently skip.*

---

### `route-map`

**Write-if:** The codebase has a defined routing layer — a place where inbound paths (HTTP,
CLI, events, or any entry mechanism) are declared and bound to handlers.

**Source:** The routing entry point(s) declared in the contract. Stack-neutral: whatever
file(s) constitute the routing layer for this project.

**Output:** One reference doc. Every inbound path mapped to its handler, auth requirement,
and any notable constraint.

```
# Route Map
*Generated by #generate-codebase-docs (route-map) on {date}. Flush-and-fill.*

| Verb | Path | Handler | Auth | Notes |
|---|---|---|---|---|
| {verb} | {path} | {Class::method or equivalent} | {guard/none} | {optional} |
```

*For non-HTTP routing (CLI commands, event listeners): use the same table with Verb replaced
by the entry-mechanism type. The deployed contract notes the stack convention.*

---

### `folder-index`

**Write-if:** The codebase has logical module folders cohesive enough that a per-folder
file listing aids navigation.

**Source:** A set of `source-glob → output-file` entries declared in the contract. Each
entry maps a glob pattern to one output document. The glob determines which source files
appear in that document. Granularity is controlled by how specific the glob is — a broad
`components/**` produces one flat doc; per-subfolder globs (`components/saturation/**`,
`components/campaigns/**`) produce separate docs per domain. The deployed contract carries
the glob entries; the generator expands them mechanically.

Glob entries are **produced at deployment**. The deployment process surveys the project
structure and generates the entries. The long-term path is script-driven generation; for now,
the deploying agent proposes patterns and the user confirms. See *Deployment Philosophy*.

**Output:** One reference doc per contract entry. Lists all files matched by the glob with
one-line purpose summaries.

```
# {glob-label}/ — Index
*Generated by #generate-codebase-docs (folder-index) on {date}. Flush-and-fill.*

| File | Purpose |
|---|---|
| {filename} | {one-line purpose inferred from source} |
```

---

### `domain-map`

**Write-if:** The codebase has identifiable classes or structured objects. The deployed
contract declares what "class" means for this stack.

**Source:** One or more domain roots declared in the contract. Each root is a source glob and
a corresponding output file. One domain-map doc is produced per root.

**Output:** One reference doc per declared root. For each class in scope: a method table.
These docs are **planning documents** — an agent should be able to plan substantial work from
them without reading source code.

**Determining access (language-agnostic):**

A method is **private** if it is not intended to be called from outside the class or module.
Determination uses this precedence, highest first:

1. Explicit language visibility keyword (`private`, `protected`, or equivalent) — always wins
2. Leading `_` prefix — a universal convention; always private
3. Internal-only docblock or annotation (`@internal`, `@private`, or equivalent)
4. AI judgment from call patterns — if the method is called only from within the same class
   body and has no plausible external use, treat as private; if called from outside, public

The deployed copy may declare a **stack-specific override** for non-idiomatic patterns. An
override replaces AI judgment for its declared scope — it is not additive. This is the escape
hatch for unusual conventions; the canonical does not prescribe language-specific rules.

**Rules:**
- Class-body methods only — do not follow inheritance chains; note the parent class name but
  do not expand it
- Return type: from declaration if present; inferred from return statement if clear; `mixed`
  if not resolvable
- Params: `name: type` if typed, `name` if not; omit default values unless semantically
  significant (a default of `null` is noise; a named constant default is signal)
- Purpose: one line; business intent not implementation; "Creates and persists a campaign"
  not "calls $this->save()"
- A class with no methods: include the class header, note "no methods defined"

```
# Domain Map — {root-name}
*Generated by #generate-codebase-docs (domain-map) on {date}. Flush-and-fill.*

## {ClassName} — {source-path}
*{One-line class purpose. Extends: {ParentClass} if applicable.}*

| Method | Access | Params | Returns | Purpose |
|---|---|---|---|---|
| {name} | public | {param list} | {type or shape} | {one line} |
| _{name} | private | {param list} | {type or shape} | {one line} |
```

### `composition-graph`

**Write-if:** The codebase has a UI component hierarchy worth mapping visually — a set of
views or pages that compose named components, where understanding the composition
relationships aids development.

**Source:** UI source roots declared in the contract (views glob + components glob). A
stack-specific **framework context file** (e.g. `context/vue-sfc-extraction.md`) is declared
in the contract; the generator reads it before extraction. This file defines how to identify
component usage in templates, resolve component names from imports, and distinguish page views
from sub-components.

**Output:** One Obsidian `.canvas` file. Unlike the other strategies, this is
**template-applied**, not flush-and-fill.

**Why template-applied:** Canvas nodes carry x/y coordinates. A generator recreating the
canvas from scratch each run produces a valid graph but an incoherent layout — positions
shift, groups scatter. The deployment session establishes the canvas layout once (groups per
page, component nodes inside groups) and stores it as `context/composition-graph-template.canvas`.
Subsequent runs refresh the edges and component membership while preserving node positions
and group structure from the template.

**Run behavior:**
1. Read `context/composition-graph-template.canvas` — the layout artifact from deployment
2. Read current source via the framework context file
3. Refresh: update component membership inside each group; update edges
4. Preserve: node positions, group boundaries, group labels
5. Surface: any page/view found in source with no corresponding group in the template —
   note in run-report; do not silently omit and do not auto-add

**New pages after deployment:** When a view is added to the source after the template was
established, the generator surfaces it in the run-report. Template update is handled by a
companion habit — `add-page-to-composition-graph` — which the user invokes to add the new
group. Front-end sessions can attach this habit to page-creation workflows. The generator
never modifies the template.

```
Output: {path}.canvas
Format: Obsidian canvas JSON — groups per page, component nodes, composition edges
```

## Deployment

*Deployment produces the output-contract. Subsequent runs are mechanical. Re-run deployment
only when the project structure changes significantly enough that the contract no longer
reflects reality.*

**Deployment philosophy:** Deployment aspires to be script-driven. AI input is limited to
steps that genuinely require judgment: write-if evaluation, initial glob pattern proposals
where project structure is ambiguous, and composition-graph initial layout. All deterministic
work — file structure survey, glob expansion, output-contract write, validation — belongs in
a script. AI is for exploration and judgment; predictable mechanics belong in code. The current
deployment steps are agentic because the automation tooling is not yet built; the shape of
each step reflects that eventual boundary.

*If deployment requires multiple sessions, record progress in `context/deployment-notes.md`
so work can be resumed without re-surveying from scratch.*

### Deployment Steps

**1 — Survey**
Read the top-level codebase structure. Identify: language(s), frameworks, entry point files,
major source folders, whether a UI layer exists. This step is mechanical; a script could
produce it. For now, the deploying agent does it and records a brief summary in
`context/deployment-notes.md`.

**2 — Strategy evaluation**
For each strategy, evaluate the write-if condition against the survey findings. Record
`enabled: true | false` and the rationale. Every decision is explicit — do not default-enable,
do not guess. This is an AI judgment step.

**3 — Glob entry generation** *(for `folder-index`, `api-surface`, `domain-map`)*
For each enabled strategy, propose source globs and output paths. The deploying agent derives
these from the survey and proposes them; the user confirms or adjusts. Future: script-generated
from a directory walk. Record confirmed entries in the contract draft.

**4 — Composition graph setup** *(if `composition-graph` enabled)*
a. Read the UI source; identify pages/views and their top-level component usage.
b. Propose a canvas layout: groups per page, component nodes inside, edges for composition.
c. Present to user; apply any layout adjustments.
d. Write the confirmed layout to `context/composition-graph-template.canvas`.
e. Write or identify the framework context file (`context/{stack}-sfc-extraction.md`).
f. Record both paths in the contract draft.

**5 — Write output-contract**
Write `context/output-contract.md` per the format above with all decisions recorded.
Future: this step is a script that takes the confirmed decisions as input and emits the file.

**6 — Validate**
Run `#generate-codebase-docs --test`. Confirm: all enabled strategies have reachable sources,
all output paths are under `reference-root`, the composition-graph template and framework
context files exist if that strategy is enabled.

**7 — Activate**
Set `deployment-status: active` in the output-contract.

## Care

**Defends:**
- The output-contract is the invariant. The generator executes it faithfully and does not
  extend beyond what the contract declares.
- Write-if decisions are deployment-time only. The running generator reads `enabled` from the
  contract and never re-evaluates conditions.
- Glob entries are strict — a file not matched by a declared glob is never indexed.
- Domain-map: class-body methods only; no inheritance chain expansion.
- Composition-graph: template node positions and group structure are never modified by a run.
  Only edges and component membership are refreshed.
- Each non-canvas output opens with a generated-by banner and flush-and-fill notice.

**Will not touch:**
- Source files (read-only, always)
- Any output file not declared in the output-contract
- Canvas node positions or group structure in the composition-graph template
- Deployment context files in `context/`

**Surfaces (notes inline, does not block):**
- A source file in scope whose purpose cannot be inferred — document the raw signal,
  note "purpose unclear — verify"
- An api-surface module with no identifiable external calls — include the section header,
  note "no external calls identified — verify"
- A domain-map class with no methods — include the class header, note "no methods defined"
- A page/view in source with no corresponding group in the composition-graph template —
  note in run-report; do not silently omit from canvas output

## Do

*Pre-flight runs first per `_generator_base`: deployment check, requirement resolution,
modifier parse.*

### Phase 1 — Contract Load

Read `context/output-contract.md`. Validate:
- `deployment-status` is `active` (warn if `draft` but do not fail)
- Every enabled strategy has its required fields present and sources reachable
- Composition-graph: `canvas-template` and `framework-context` paths resolve

If any enabled strategy is missing a required field:
```
FAILED — output-contract incomplete: {strategy} missing {field}
```

Build the run plan: enabled strategies in canonical order (api-surface → route-map →
folder-index → domain-map → composition-graph), or in contract-declared order if present.
Apply modifier filtering.

### Phase 2–N — Strategy Execution

Execute each strategy in run-plan order. Each is one phase in the run-report; phase number
reflects position in the run plan, not a fixed integer. A disabled strategy is not a phase.

- **api-surface:** read each declared source file; extract all outbound calls; assemble the
  combined doc (flush-and-fill)
- **route-map:** read the routing layer; extract all routes and handler bindings; write the
  route-map doc (flush-and-fill)
- **folder-index:** for each glob entry, expand the glob, extract file list and infer
  purposes, write the index doc (flush-and-fill)
- **domain-map:** for each root, read the source glob; extract class and method tables per
  the access rules above; write the root's doc (flush-and-fill)
- **composition-graph:** read framework context file; read source; read canvas template;
  refresh edges and membership; write output canvas (template-applied)

### Phase N+1 — Run Log + Output Declaration

Append one row to `run-log.md` in the deployed generator's folder (create if absent;
append-only, never flushed):

| Date | Modifier | Strategies run | Confidence range | Notes |
|---|---|---|---|---|

Output Declaration: lists every file written this run.

After this phase, the `run-report` habit fires automatically (writes
`_Claude/audits/generate-codebase-docs-run.md`, flush-and-fill).

## Design Notes

*Open items carried forward from authoring. Resolved items removed.*

1. **Functional / non-OO modules in domain-map** — the method-table format assumes a class.
   Purely functional modules (utility files, helper collections) have no class concept. The
   likely answer is a `## {module} (helpers)` section variant with the same table but
   organized by file rather than class. This produces its own question: where does helper
   output go — as part of domain-map, or as a separate strategy? Open question; punt for now.
   Instinct: a `helpers` block in the domain-map doc, assembled separately.

2. **Deployment automation tooling** — the deployment steps are agentic today but designed
   to migrate to scripts. The boundary is clear (judgment steps stay AI; mechanics move to
   code). This is a future workstream, not a blocker for the canonical.

3. **api-surface / route-map cross-reference enrichment** — now that api-surface runs first,
   the route-map could flag routes that have no corresponding frontend call in the api-surface
   doc. This would be a useful gap-identification feature. Requires the route-map phase to
   read the api-surface output. Not in scope for the initial canonical; worth noting as a
   future enhancement.
