author-reference — Habit

Inherited by all lenses via _lens_base. Defines the shape of a reference artifact — the durable, project-level store of truth. The mechanics of the HTML itself are kcd-document-protocol; this habit is the reference-specific section vocabulary and the rules that keep one durable.

Why

Authoring a new reference, migrating one to the HTML substrate, or auditing an existing one. A reference is durable state — the settled how-it-works, not a plan or a todo. (A forward-looking observation is a note, which keeps its open sections; a procedural recipe is a how-to, which keeps its own step shape.)

Action

author or audit the reference using the canonical section vocabulary below — overview · free-named topical · rules (union) · references (union) — and keep it durable

Explanation

A reference is the durable, project-level store of truth — the settled how-it-works, not a plan or a todo (a forward-looking observation is a note, which keeps its open sections; a procedural recipe is a how-to). The HTML mechanics are kcd-document-protocol; this habit is the reference-specific section vocabulary and the rules that keep one durable and dense (dense reference graphs let agents find things).

Frontmatter

Exactly the universal set: name · description · type · status · schema-version · author · updated. No category / domain field — the folder encodes the category. Author is the block form Name <email>. Preserve the original updated date when migrating — a mechanical move does not re-date the content.

Sections — the vocabulary

A reference has no Know/Care/Do region; sections sit directly under <article data-kcd="reference">. Each section is a typed property the parser extracts (an authoring surface), not a freeform blob.

  • overview (additive) — one prose block: what it is and why it exists, with the mental model / shape / layers folded in. The one-paragraph orientation.
  • Topical sections (free-named, additive) — one named property per doc-specific mechanism (the-truck, guard-chain, the-vault…). The irreducible body. Trim hard: each earns its place; cut the tangential and the stale.
  • rules (data-kcd-merge="union") — short, authoritative this-not-that statements, each a data-kcd-slot with a data-kcd-field="rule" cell (the rule text stays prose). Rules cover only the code the reference idiomatically governs. Union so a lens dredging several references stacks their rules into one deduped view.
  • references (data-kcd-merge="union") — one merged What/Where/Why faux-table (the index-format shape). Source files, links, see-also, and relations all fold in here — a reference IS a relation (info + a path to trawl). where is a real <a href> (path or url). Dense reference graphs are the point: they let agents find things.

Human aids (cheat-sheet, worked example, figure, a <details> rationale accordion) carry data-kcd-audience="human" — kept for the visual read, stripped from the AI read.

Rules

Rule
Durable state only — drop deferred. Deferred / open / not-built / todo content is not a reference section. Leave it in the source or file it on the owning lens; a reference carries only what is settled.
Prose is canonical, not chrome. Load-bearing body prose lives inside a data-kcd-section (merged into overview or surfaced as its own topical property). Only genuinely decorative prose is dropped.
Illustrative tables are chrome; canonical rows are faux-tables. A descriptive table carrying no extracted data is a plain <table> (parser ignores it). Anything with data-kcd-slot/-field is a styled-div faux-table — never a real <table> (protocol §3).
Files, links, and relations are one property. Do not scatter a "Files" list, a "Links" list, and a "Related" list — merge them into the single references section.
The folder is the category. Location encodes domain; never restate it as a frontmatter field.
