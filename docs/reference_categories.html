Reference Categories — Framework

Framework meta-knowledge. Not auto-loaded. Read when adding a reference, or when deciding where a piece of knowledge belongs.

The rule

A KCD codebase has one shared reference store: _Claude/references/. It is the project-level store of truth — facts about the code, conventions, and domain that lenses point at via their Know block.

References are categorized, not flat. Every reference file lives in a category folder. A bare file at references/ root (other than nav-index.html) is a structural violation — it has no declared category, so no one knows what kind of truth it is.

Lens-specific vs. shared. references/ holds material useful across lenses. Material that travels only with one lens lives in that lens's own context/ (lenses/{name}/context/), not in the shared store — keep the library shared and findable, keep one-lens material local.

kcd/ is the single source for everything that is not a reference: framework meta-docs, templates, habits, generators, analyzers. If a document explains how the document system works, it belongs in kcd/. If it states a fact about the project, it belongs in a references/ category.

Standard categories

These four are the KCD-recommended top-level categories. A project may add more, but should justify each addition — categories are a taxonomy, and a taxonomy with twenty buckets classifies nothing.

_Claude/references/
  nav-index.html            catalog of every reference and its category
  style-guides/             how code should be written (formatting, naming, scope)
  patterns/                 reusable structural solutions (how to build an X)
  domain/                   business and data facts (schema, vocabulary, rules)
  features/                 hand-authored feature specs and promoted plans (deploy empty)
  notes/                    hand-authored session observations and references (deploy empty)
  codebase/                 generated navigation maps — generator-only, flush-and-fill
    {sub-project}/          one folder per code sub-project
CategoryHoldsAuthored byVolatility
style-guides/Language and formatting conventionsHuman, refined over timeLow
patterns/"How we build an X here" structural templatesHumanLow
domain/Schema, business vocabulary, invariantsHuman + generatedMedium
features/Feature specs, component records, promoted plansHuman (session)Medium
notes/Session-produced observations and authored codebase notesHuman (session)Medium
codebase/Route maps, controller maps, component mapsGenerated by a generatorHigh — regenerated

codebase/ is the only category written by a generator — it is generator-only and flush-and-fill. No hand-authored file ever lives here. It is split per sub-project so a multi-repo codebase does not collide names (backend/routes.md vs frontend/routes.md).

features/ and notes/ deploy empty and fill as sessions produce authored references. Both are linked to lenses via Know tables so they are discoverable. A future subfolder convention may be needed when either grows large, but flat is fine until then.

Example instantiation

_Claude/references/
  style-guides/   global.md  comments.md  php.md  vue.md  css.md
  patterns/       controller.md  sql.md  vue.md
  domain/         database.md  domain_concepts.md
  codebase/
    backend/      routes.md  controllers.md  data_map.md
    frontend/     routes.md  views.md  stores.md  api.md  components.md

Filenames are lowercase. The category folder already carries the "what kind of thing" signal, so the filename only needs the specific subject.

Where this is referenced from

  • kcd/kcd_framework.html links here under "Reference categories."
  • Lens Know blocks point at specific files inside categories, never at the references/ root.
  • references/nav-index.html is the per-project catalog — it lists every file and is the human-facing "where is X" lookup.
