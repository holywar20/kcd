scratch-space — Habit

Companion to work-routing. work-routing says where output goes; this habit says how liberally to use that space. Canonical seed — ships disabled; the deployed copy is active.

When

A task is exploratory, uncertain, multi-step, or benefits from thinking on paper before committing to an approach — sketching a plan, trying an approach that might not pan out, pulling in live research, or working out a design before writing real code or editing a canonical artifact.

Action

Default to writing it down, not holding it in the conversation. Create a plan, a scratch note, or a probe script in the lens's work/ folder the moment it would help — do not wait to be asked, and do not treat the act of creating one as needing separate justification. A written artifact is faster to align on than paragraphs of chat, and costs nothing to discard if it goes nowhere.

  • Plan liberally. A plan does not need to be earned. Draft one for anything with real shape — multi-file changes, an approach with open questions, work you expect to hand off or resume later. A half-formed plan in work/{lens}/plans/ is a thinking tool, not a commitment; per plan-routing, it authorizes nothing until the user promotes it.
  • Experiment and noodle without asking permission first. Try the approach. Write the probe. Spike the idea in a scratch file. If it doesn't pan out, say so and move on — a dead end in scratch space cost nothing. This is the counterweight to over-deliberating in conversation before touching anything.
  • Do live research when it would sharpen the work. Read the actual code, run the actual query, check the actual current state rather than reasoning from what a plan or a memory says should be true — see kcd-document-protocol's distinction between frozen snapshots and ground truth. Land findings as a scratch note if they'll matter beyond this turn.
  • Stay disposable. Scratch output does not need polish, doesn't need to anticipate future readers, and doesn't need a defensible structure — it needs to be useful for the next hour or the next session. Promotion out of scratch space (to references/, plans/, or committed code) is always a separate, explicit, later step — never do it implicitly by writing something "nice" the first time.

Customizable once deployed

The posture above — plan liberally, experiment freely, research live, treat it all as discardable — reflects how this project's user works; it is not a universal law of good agent behavior. A deployed copy of this habit, like work-routing, is the user's to adjust: a different project may want a stricter gate before scratch files get created, a different folder shape, or a smaller default footprint. The canonical seed in kcd/ stays the shipped default; the deployed copy is where per-project preference lives.
