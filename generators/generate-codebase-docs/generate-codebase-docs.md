# generate-codebase-docs — Generator (canonical)

*Generic across any multi-area codebase. The deployed copy solves the requirements below with
project-specific values. Interpretive — it summarizes ownership, purpose, and structure from
source, not just mechanical extraction. Sonnet, not haiku.*

Base rules: [_generator_base](_Claude/kcd/generators/_generator_base.md) — composition model,
requirement resolution, fail behavior, output conventions, reference wiring, modifiers.

---

## Parameters

Output: the `reference-root` requirement (solved below), one subtree per area

*`name` / `#{name}` derive from the filename; `model` and `base` live in frontmatter.*

---

## Modifiers

*Per `_generator_base`: `--`-prefixed, globally unique, registered in the Modifier Registry, may
be omitted. One modifier per area declared in `doc-areas`. No modifier = all areas. Unknown
modifier = fail. `--test` is inherited from `_generator_base`.*

| Modifier | Effect |
|---|---|
| `--{area}` | Regenerate only that area's doc set (one modifier per `doc-areas` entry; solved in the deployed copy) |

---

## Requirements

*Declared generically; solved in the deployed copy. See `_generator_base` for pre-flight
resolution and the fail rule.*

| Name | Kind | Description |
|---|---|---|
| `doc-areas` | Input | The named areas this generator can target. Each area declares: its modifier, its source root/globs, and the ordered set of reference docs it produces (filename + what each doc covers). |
| `reference-root` | Input | Base directory under which each area's doc set is written. |
| `reference-lenses` | Input | Per-area: the lens(es) whose Know table must point to that area's docs. |

---

## Care

**Defends:**
- Each doc is flush-and-fill — fully regenerated from source each run, never hand-patched.
- The doc set per area is **closed** — only the declared docs are written. Files in the same
  output subtree that are *not* in `doc-areas` are never touched (hand-authored references live
  alongside generated ones).
- Summaries are interpretive but evidence-bound — ownership/purpose claims trace to source.

**Will not touch:**
- Source files — read-only.
- Any output file not named in the invoked area's `doc-areas` doc set.
- Lens content other than the Know-table rows for the area's docs.

**Surfaces (notes inline, does not block):**
- A source element whose purpose cannot be inferred — document the raw signal, flag "purpose unclear — verify".

---

## Do

*Pre-flight runs first, per `_generator_base` — deployment check, requirement resolution,
modifier parse.*

### Phase 1 — Area Resolution

Parse the invocation modifier. If one is present, select that area from `doc-areas`; if absent,
select all areas. An unknown modifier fails the run per `_generator_base`.

### Phase 2 — Source Read (per selected area)

Read the area's source root/globs. Build the structural picture the area's docs require —
routes, handlers, modules, components, data tables, whatever the area's declared doc set covers.

### Phase 3 — Doc Generation (per selected area, per declared doc)

For each declared doc, produce its content interpretively: structure tables plus one-line
ownership/purpose summaries grounded in the source. Each doc opens with a generated-by banner:

```
*Generated by #generate-codebase-docs ({area}) on {run date}. Flush-and-fill — do not hand-edit; rerun the generator.*
```

### Phase 4 — Write (flush-and-fill)

Write each declared doc under `reference-root`, destroying and recreating it. Touch no other file.

### Phase 5 — Lens Wiring

For each lens in the area's `reference-lenses`, ensure its Know table has a `What | Where | Why`
row per generated doc — add if missing, correct if stale. Skipped under `--test`.

### Phase 6 — Output Declaration

Writes the invoked area(s)' declared docs under `reference-root` (flush-and-fill) and edits the
Know tables of the area's `reference-lenses`. No other output.
