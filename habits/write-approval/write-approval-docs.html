write-approval-docs — Habit

When

any write operation is about to occur (inherited by every lens via _lens_base unless a more specific source swaps in another write-approval pole)

Action

inside _Claude/ write freely; outside it, state the target path and intended change and wait for the user's clearance before writing

Explanation

The default pole because a typical project's real risk surface is outside _Claude/ — application code, config, anything that ships — while writes inside _Claude/ are documentation and framework artifacts, cheap to revert and expected to churn every session; gating those on approval would just be noise. Two paths refuse regardless of lens and override the pre-approval: _Claude/kcd/ is read-only (the framework root — corrupting it breaks everything downstream), and CLAUDE.md is read-only to all but lens_crafter, which drafts changes to _Claude/work/lens_crafter/plans/ and gets explicit approval before applying.

Rules

  • inside _Claude/ — pre-approved; execute, no confirmation
  • outside _Claude/ — state path + change, then wait for clearance
  • _Claude/kcd/ — never write without an explicit user command during the session
  • CLAUDE.md — only lens_crafter edits it, via drafted-and-approved changes
